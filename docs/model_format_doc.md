## TECHNICAL SHEET — `.simmodel` FORMAT SPECIFICATION

# 1. Overview

The `.simmodel` format is a data container designed to encapsulate an entire parametric simulation model based on a multidimensional grid and a dynamic rule system.

The format must satisfy the following constraints simultaneously:

* **Execution performance**: direct memory access, low parsing cost
* **Modularity**: strict separation between data, logic, and compiled artifacts
* **Mutability**: ability to modify the model and logic at runtime
* **Extensibility**: future addition of new backends or data types
* **Portability**: platform independence

The file is designed as a **multi-representation container**, where the following coexist:

* an **editable** version
* a **runtime-optimized** version
* **compilation caches**

# 2. Physical File Format

## 2.1 Container Type

The `.simmodel` file is a compressed archive (ZIP or equivalent), organized like an embedded file system.

This choice allows:

* coexistence of multiple representations of the same object
* partial reading (streaming possible)
* simple version and extension management

## 2.2 Internal Organization

Typical structure:

```id="yfjq5t"
/model.bin
/logic.ir
/layout.bin
/metadata.json
/version.json

# Optional
/model.json
/logic.opt.bin
/logic.cpu.bin
/logic.gpu.spv
/assets/...
```

Each internal file has a strictly defined, independent role.

# 3. Data Representation (Model)

## 3.1 Main Binary Format (`model.bin`)

The `model.bin` file is the runtime source of truth.

### Design Constraints:

* Structured binary encoding (FlatBuffers recommended)
* Direct access without full deserialization
* No deeply recursive structures
* Compact indexing (integers rather than dynamic references)

### Logical Structure:

The model is an enriched directed graph coupled with a grid definition.

#### Nodes

Represent functional model units (operations, sources, transformations).

Typical attributes:

* unique identifier
* type (mathematical operation, data source, etc.)
* scalar or vector parameters
* references to variables

#### Edges

Define dependencies between nodes:

* data flow
* compute dependencies

Each edge is defined by:

* source
* destination
* relationship type

#### Variables

Describe the state manipulated by the simulation:

* name
* type (scalar, vector, tensor)
* scope:

  * cell (per grid element)
  * global
  * boundary
* initial value

#### Grid Definition

* number of dimensions
* size per dimension
* topology type (Cartesian, hexagonal, custom)
* boundary conditions

## 3.2 Editable Representation (`model.json`)

Text version of the model used exclusively by the editor.

It enables:

* human inspection
* external modification
* interoperability

It is never used directly at runtime.

# 4. Logic Representation

## 4.1 Intermediate Representation (`logic.ir`)

Model logic is defined as an intermediate representation (IR), independent of any architecture.

### Objectives:

* decouple logic from the execution backend
* enable static optimizations
* serve as a single source for CPU and GPU compilation

### IR Nature:

* tree structure (AST) or acyclic graph (DAG)
* explicit atomic operations (add, mul, load, store, etc.)
* no semantic ambiguity

### Properties:

* strict typing
* no implicit side effects
* explicitly modeled dependencies

## 4.2 Optimized IR (`logic.opt.bin`)

Transformed version of `logic.ir` after passing through an optimization pipeline.

### Typical Transformations:

* constant propagation
* dead-code elimination
* operation fusion
* algebraic simplification
* dependency reduction

This version is used to reduce dynamic compilation costs.

## 4.3 Compiled Artifacts

### CPU (`logic.cpu.bin`)

Native machine code generated by JIT compilation (e.g., LLVM).

Characteristics:

* architecture-dependent
* non-portable
* invalidated when the model or engine changes

### GPU (`logic.gpu.spv`)

Compiled compute shader (SPIR-V).

Characteristics:

* massive parallel execution
* requires GPU-compatible memory layout
* strong constraints on compute structure

# 5. Memory Organization (`layout.bin`)

## 5.1 Role

Precisely describe how simulation data is laid out in memory.

This is a critical performance factor.

## 5.2 Recommended Model

Structure of Arrays (SoA):

* each variable is stored in a contiguous buffer
* optimized sequential access
* SIMD- and GPU-compatible

## 5.3 Contents

* variable list
* memory offset
* alignment
* stride
* mapping CPU/GPU

## 5.4 Constraints

* explicit memory alignment
* no implicit padding
* strict consistency with `model.bin`

# 6. Metadata and Versioning

## 6.1 Metadata (`metadata.json`)

Non-critical execution information:

* model identification
* author
* description
* tags
* history

## 6.2 Version (`version.json`)

Used to manage compatibility:

* format version
* minimum required engine version

The engine must reject or migrate incompatible formats.

# 7. Execution Pipeline

## 7.1 Loading

* archive opening
* version validation
* loading binary structures (`model.bin`, `layout.bin`)

## 7.2 Logic Initialization

* load `logic.opt.bin` if available
* otherwise load `logic.ir`
* apply optimizations

## 7.3 Compilation / Backend Selection

Depending on execution mode:

* CPU :

  * load native cache or perform JIT compilation
* GPU :

  * load shader or generate from IR

## 7.4 Execution

* memory allocation according to `layout.bin`
* grid iteration
* rule application through selected backend

# 8. Change Management

## 8.1 Structure Modification

* requires regeneration of `model.bin`
* invalidates caches (`logic.*.bin`)

## 8.2 Logic Modification

* update `logic.ir`
* partial or full recompilation

## 8.3 Interactive Mode

* use of an IR-based interpreter
* deferred compilation for optimization

# 9. Performance Constraints

* no dynamic allocations in critical loops
* prioritize sequential memory access
* minimize conditional branches
* maximize vectorization (SIMD)
* limit CPU ↔ GPU transfers

Memory layout is the dominant performance factor, ahead of language or backend choice.

# 10. Extension Principles

The format must allow:

* adding new internal files without breaking compatibility
* coexistence of multiple backends
* introduction of new variable types
* support for new architectures

Any extension must remain optional and ignorable by older versions.

# 11. Summary

The `.simmodel` format relies on strict separation:

* **Data**: `model.bin`
* **Logic**: `logic.ir`
* **Optimizations / execution**: `logic.*`
* **Memory performance**: `layout.bin`

This decoupling enables:

* highly optimized execution
* strong modification flexibility
* compatibility with multiple execution strategies
* long-term scalability

# 12. Formal IR Specification (Intermediate Representation)

## 12.1. Grammar and Structure (Typed AST)
IR is a strictly typed Abstract Syntax Tree (AST) or a Directed Acyclic Graph (DAG) of operations.
*   **Data Nodes (Leaves):**
  *   Load(VariableID, OffsetX, OffsetY): Read a variable at a relative position (0,0 = current cell).
  *   Constant(Value, Type): Literal constant (e.g., 9.81, f32).
  *   GlobalLoad(GlobalVarID): Read a unified global variable.
*   **Mathematical Nodes:**
    *   Add, Sub, Mul, Div, Mod, Pow.
    *   Abs, Min, Max, Clamp, Sign.
  *   Transcendental functions: Sin, Cos, Exp, Log.
*   **Spatial Operators:**
  *   Gradient(VariableID, Axis): Compute local gradient.
  *   Laplacian(VariableID): Apply Laplacian operator.
*   **Restricted Flow Control:**
  *   Select(Condition, TrueBranch, FalseBranch): Ternary (vectorizable if-else).
  *   *Absolute prohibition* of unbounded loops (dynamic while/for) to guarantee O(1) compute time per cell and facilitate SIMD/GPU vectorization.
*   **Output Nodes (Roots):**
  *   Store(VariableID, Value): Write into the target's *next* buffer (double buffering).

## 12.2. Strict Type System
IR has no implicit casting. All conversions (f32 to f64, i32 to f32) must be explicit Cast(Value, TargetType) nodes.
*   **Scalars:** bool, i32, u32, f32, f64.
*   **Vectors (up to 4 dimensions):** vec2, vec3, vec4 (f32 or f64). Arithmetic operations on vectors are applied *component-wise*.

## 12.3. SI Unit Conventions (International System)
World-Simulator encourages explicit SI base-unit expressions in model files and reports derived-unit aliases through validation warnings.

The seven SI base units remain the canonical basis for authoring and internal expression:

* Mass: `kg`
* Length: `m`
* Time: `s`
* Electric current: `A` (Ampere)
* Temperature: `K` (Kelvin)
* Amount of substance: `mol`
* Luminous intensity: `cd` (Candela)

Alias detection currently covers common derived units such as `Pa`, `W`, `J`, `N`, `Hz`, and `V`. When these appear, the validator suggests explicit base-unit expressions such as `kg/(m*s^2)` or `kg*m^2/s^3`.

The editor may present friendlier labels in the UI, but the stored model representation should remain explicit and deterministic.

# 13. IR Optimization Pipeline

Before compilation, IR goes through a series of optimization passes that transform `logic.ir` into `logic.opt.bin`.

## 13.1. Pass 1: Semantic and Dimensional Validation
* Formal type checking rejects invalid operations such as scalar-vector mixing.
* Dimensional checks verify that unit expressions are compatible where enabled.

## 13.2. Pass 2: Constant Folding
* All subtrees composed only of constants are evaluated at compile time.

## 13.3. Pass 3: Algebraic Simplification and Dead-Code Elimination
* `Mul(X, 1) -> X`
* `Add(X, 0) -> X`
* `Mul(X, 0) -> null`
* Nodes whose results never reach a `Store` node are removed from the graph.

## 13.4. Pass 4: Access Pattern Analysis
* The optimizer inspects offsets requested by `Load` nodes.
* Repeated neighborhood access can inform cache-friendly layouts or future backend hints.

# 14. CPU and GPU Compilation Rules

World-Simulator currently runs simulation logic on CPU for reproducibility.

## 14.1. CPU Compilation
* Memory layout follows Structure of Arrays for field buffers.
* SIMD-friendly lowering is preferred for hot loops.
* The scheduler and checkpoint logic remain CPU-resident.

## 14.2. GPU Artifacts
* `logic.gpu.spv` is best treated as a future or optional backend artifact.
* If a GPU execution path is added later, it must be explicit, opt-in, and validated against the CPU baseline.
* Visualization shaders are separate from simulation logic and do not change runtime determinism.

## 14.3. Fallback Rule
If a backend cannot preserve the runtime contract, the engine must fall back to the CPU path rather than silently relaxing determinism.
